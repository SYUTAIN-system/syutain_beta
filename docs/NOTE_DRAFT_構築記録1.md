# 【構築記録 #1】設計書を読ませた日。100個のバグと、「動いている」の嘘。

前回、#0で「設計が終わった」と書いた。
あとはやるだけだ、と。

実際にやった。
Claude Codeに25,000行の設計書を読ませて、「Step 1から始めて」と言った。

そして、100個以上のバグが出た。

---

## スケジューラーが死んでいた

最初の数時間は順調に見えた。
FastAPIが起動し、Next.jsが200を返し、NATSが接続される。
ダッシュボードにアクセスすると、ノード4台がすべて「healthy」と表示されていた。

動いている。…ように見えた。

だが、スケジューラーが75回再起動に失敗して完全に死んでいた。
SNS投稿も、PostgreSQLバックアップも、ヘルスモニタリングも、情報収集パイプラインも。
19個以上のジョブが全部止まっていた。

原因は `timezone=JST` という6箇所のコード。
Pythonの変数 `JST` がどこにも定義されていなかった。
スケジューラーのデフォルトタイムゾーンが既に `"Asia/Tokyo"` に設定されていたから、この6箇所を消すだけで全部直る。

**たった6文字を消すだけで直る問題が、全てを止めていた。**

正直、笑えなかった。
「動いているか確認しろ」とCLAUDE.mdに書いたのは僕だ。
確認した。表示は正常だった。でもスケジューラーのプロセスは落ちていた。

「確認した」と「正しく確認した」は別物だ。

---

## 70個のバグ

スケジューラーを直した後、システム全体に総合デバッグをかけた。
6つの監査エージェントを並列で走らせて、全52個のPythonファイル、全18個のTypeScriptファイル、全32テーブルを精査した。

出てきたバグは70件。
内訳はこうだ。

- CRITICAL（システム停止レベル）: 12件
- HIGH（重要機能の障害）: 18件
- MEDIUM（品質・設計問題）: 25件
- LOW（保守性）: 15件

SQLインジェクション。パスワードの平文保存。Emergency Killが一度発動すると全ゴールが永久停止するバグ。SNS投稿が承認フローを完全にバイパスしている設計。

1つ1つは「なんでこうなってるんだ」という問題だった。
でも、全部に共通するパターンがあった。

**設計書に書いてあるのは「こう動くべき」であって、「こう動いている」ではない。**

コードが存在することと、コードが正しく動くことは、全く別の話だ。

---

## 「active」の嘘

最も教訓になったのは、Phase 4の話だ。

リモートワーカー（BRAVO/CHARLIE/DELTA）の状態を確認した。
「全3ノードactive、プロセス存在」と報告された。

だが後日、別の方法で確認したら全ノード「inactive」だった。

何が起きていたか。最初の確認では `syutain-worker-*` というワイルドカードで検索していた。実際のユニット名は `syutain-worker-bravo` のような具体名だった。パターンが合わなかっただけで、存在しないように見えた。

2回目の確認では具体名で検索したから「active」と出た。

どっちも「確認した」のに、結果が逆。
確認方法が間違っていたら、確認したことにならない。

これはAIの問題じゃない。僕の問題だ。
「確認しろ」と指示する人間が、何をもって確認とするかを定義しなければ、AIは「確認しました」と答えるだけだ。

## Brain-αと設計書の乖離

70件のバグを直した後に、Brain-α融合の検証をした。

Claude Code ChannelsでDiscordから対話できる。Hooksで安全装置が動く。精査サイクル8Phaseが回る。
…と設計書には書いてある。

調べてみたら、設計書v2とClaude Codeの実際の仕様に乖離があった。

Dispatchは「Brain-αのインターフェース」ではなくCoworkのリモートアクセスだった。
Hooksのブロックはexit code 1ではなく2で行う仕様だった。
Stopフックは「セッション終了時」ではなく「応答完了のたびに」発火する。

設計時にリリースされたばかりの機能を「こうなるはず」で書いていた部分が、実際の仕様と食い違っていた。

これも「設計なき実装」の変形版だ。正確には「推測に基づく設計」。

## Nemotronが考え始めた

面白い発見もあった。

Nemotron-Nano-9B-v2-Japanese。NVIDIA製の日本語特化LLM。
BRAVOのRTX 5070にロードしてある。

このモデルに `/think`（推論モード）があるのだが、動かなかった。
前のセッションで「Nemotronはthink非対応」と判断して、無効化していた。

でも調べ直したら、対応していた。
HuggingFaceのモデルカードにもOllamaのCapabilitiesにも `thinking` と書いてある。

問題はModelfileのテンプレート設定だった。
CHARLIEのOllamaを0.18.0から0.18.2にアップデートし、BRAVOのModelfileをコピーして再作成。

修正したら、thinking=2,948文字の推論トレースが出た。
品質検証がローカルLLMの推論モードで動く。コスト0円。

**コードを1行も書いていない人間が、1日で100個のバグを直した。直したのは僕じゃない。僕が書いたのはプロンプトだ。**
