# コードを一行も書けない人間が、51,672行の自律AIシステムを作った話——壊れ続けた16日間の全記録

コードが書けない。変数って何？レベル。

本業は映像制作。VFX、動画編集、カラーグレーディング、撮影、ドローン。8年間、VTuber業界で裏方をやってきた。プログラミングとは縁がない人間だった。

2025年後半、Claude Codeと出会った。
「AIにコードを書かせて、自分は設計だけする」——そんな話を聞いて、試してみた。

半年後。手元には51,672行のPythonコードと、4台のPCで24時間動く自律AIシステムがあった。

これは成功の話じゃない。壊れ続けた16日間の、全記録だ。

---

## なぜ作ったか

AIツールが溢れてる。ChatGPT、Claude、Gemini、Midjourney。でも、どれも「使う」止まりだった。

僕が知りたかったのは、もっと根本的なことだ。

AIは、事業を「回せる」のか。
人間が寝ている間も、情報を集め、記事を書き、SNSに投稿し、品質をチェックし、失敗から学び、次の行動を自分で決められるのか。

答えを知るには、作るしかなかった。

## SYUTAINβとは

4台のPC（Mac mini M4 Pro、RTX 5070マシン、RTX 3080マシン、GTX 980Tiマシン）をTailscale VPNで接続し、NATSメッセージングで通信する分散AIシステム。

6体のAI「役員」が動いている。

| Bot | 役割 | マシン | LLM |
|-----|------|--------|-----|
| CORTEX | CEO | ALPHA (Mac mini) | qwen3.5:4b |
| FANG | CSO | BRAVO (RTX 5070) | qwen3.5:9b |
| NERVE | COO | BRAVO | qwen3.5:9b |
| FORGE | CTO | CHARLIE (RTX 3080) | qwen3.5:9b |
| MEDULLA | 副CEO | DELTA (GTX 980Ti) | qwen3.5:4b |
| SCOUT | 情報収集 | DELTA | qwen3.5:4b |

CORTEXが10分おきに意思決定する。MEDULLAが30分おきに巡回する。FANGが毎晩21時にKPIレポートを出す。

85%をローカルLLM（Ollama qwen3.5）で処理して、月854円。コンビニのコーヒー3杯分で、6体のAIエージェントが動く。

## 壊れた話をする

ここからが本題だ。SYUTAINβは「動くシステム」じゃない。「壊れ続けるシステム」だ。壊れ方が面白い。

### 3月19日：初回クリーンスタート

エラー0件。全ノード接続。NATSクラスタ形成。初ハートビート送信。

「もう大丈夫だ」と思った。大丈夫じゃなかった。

---

### 3月20日：最初のカスケード障害

3つ同時に壊れた。

**1. 依存パッケージ未インストール**

リモートの3台に`genai`、`openai`、`anthropic`が入ってなかった。設計書にはAPI呼び出しが書いてある。コードにも書いてある。でもpip installしてなかった。全LLM呼び出しが`ModuleNotFoundError`。

**2. SNS暗号化の罠**

SNS投稿モジュールがブラウザのCookieを復号しようとした。macOSのChromium Cookieはキーチェーンに紐づいてる。別マシンにコピーしたプロファイルじゃ復号できない。これは仕様。でも知らなかった。

**3. DELTA 404**

DELTAのワーカーが全リクエストに404を返す。原因：ルーティングの登録順。キャッチオールパスが個別ルートを飲み込んでた。FastAPIのルート定義順序の罠。

この日学んだこと：「動く」と「全部動く」は全然違う。

---

### 3月23日：70バグ、1セッション、12時間

Claude Codeとの1セッションで70個のバグを潰した。12時間ぶっ通し。

この日の最大の成果は **Brain-α融合**。

それまで、自律ループは「目標を受けて、計画して、実行して、検証する」だけだった。僕が誰で、何を大事にしていて、何を絶対にしないか——そういう人格データを一切参照してなかった。

Brain-αは547件のペルソナメモリを持つ。僕の価値観、タブー、文体の癖、好みの表現。これを自律ループの認識フェーズに組み込んだ。

結果：AIが「島原大知」として振る舞い始めた。完璧じゃない。でも、前日までの「誰でもない誰か」よりはるかにマシになった。

ローカル推論比率がこの日初めて **83%** を超えた。ルーティングロジックがやっと機能し始めた。

---

### 3月25日：最悪の日——44エラー、システム全停止

この日の話は長くなる。長くなるけど、全部書く。SYUTAINβで一番学びが大きかった日だから。

**始まり：BRAVOタイムアウト**

BRAVOに送ったタスクがタイムアウトした。エグゼキュータがリトライした。またタイムアウト。プランナーが新しいプランを立てた。新しいプランもBRAVOに送った。BRAVOにしかない機能だったから。またタイムアウト。

ここまでは想定の範囲内。リトライとフォールバックは設計してある。

**問題はセマンティックループ検知器**

ループ検知器がタイムアウトの反復を検知した。正しい。これはループだ。検知器は正しく動いた。

でも、検知器は **グローバル** だった。ゴール単位じゃなかった。

ゴールAのBRAVOタイムアウトループを検知して、セマンティックループ検知器が発動した。発動すると、**全ゴールが停止する**。ゴールB、C、D——BRAVOと何の関係もないゴールも全部止まった。

この日、セマンティックループ検知器は **15回** 発動した。

15回。

発動するたびにシステム全体が凍結する。6体のAIボットが何もできずにアイドル状態。電気だけ消費して、出力ゼロ。

**44エラー**。1日で。

**修正**

`SemanticLoopDetector`をゴール単位に再設計した。各ゴールが独自のアクション履歴、独自の類似度ウィンドウ、独自のトリガー閾値を持つ。1つのゴールのループが他のゴールに影響しない。

パラメータ調整じゃない。リトライでもない。アーキテクチャの構造変更。

この日の教訓：**安全装置が最大の危険になり得る**。暴走を防ぐために作ったガードが、システム全体を止めた。ガードは正しく動いた。スコープが間違っていた。

---

### 3月28日：6つの致命バグ + ハーネスエンジニアリングの誕生

**perceiver.pyのasyncioスコーピングバグ**

認識エンジンが`asyncio.gather()`で14項目を並列実行していた。その中の1つのコルーチンが、外部スコープの変数を参照していた。その変数は、コルーチンが実行される頃にはガベージコレクションで消えていた。結果：ランダムに出現する`UnboundLocalError`。30分正常動作してから突然クラッシュ。再現性なし。

発見に4時間かかった。

**plannerのlist/dictクラッシュ**

プランナーはタスク割り当てをdict型のリストで期待していた。タスクが1件のとき、LLMがリストではなくdictを直接返した。`for task in plan_output`がdictのキーを反復する。`"node_name"`（文字列）にタスクが割り当てられる。サイレント破壊。プランは動いているように見えて、全く違うことをしていた。

**DBスキーマ不一致**

`db_init.py`のDDLと`executor.py`のINSERT文で、カラムの順序が違っていた。PostgreSQLの位置指定INSERTは名前でマッチしない。データが間違ったカラムに入る。

この日、方法論に名前をつけた。**ハーネスエンジニアリング**。

考え方はシンプルだ。

```
エラー発生
  → 即時修正（止血）
  → 根本原因分析（なぜ起きたか）
  → ガードレール構築（そのエラーの種類全体を不可能にする）
  → CLAUDE.mdに追加（AIが再導入しないようにする）
  → エピソード記憶に記録（システムが学習する）
```

perceiverバグ → スコーピングテスト追加。plannerバグ → 型正規化レイヤー追加。DBバグ → スキーマvsクエリ自動監査追加。

「同じバグを二度出さない」じゃなく、「同じ種類のバグを二度出さない」。個別修正じゃなく、構造的予防。これがハーネスエンジニアリング。

### 3月29日：3日間、誰も気づかなかった

DELTAのダッシュボードを見た。全部グリーン。BRAVOもグリーン。CHARLIEもグリーン。

実際は、**3台とも3日前から停止していた**。

ダッシュボードはキャッシュデータを表示していた。最後に取得した状態。今の状態じゃない。

3日間。リモートワーカー3台が全停止。アラートなし。エラーなし。システムはALPHAだけのローカル推論に静かにフォールバックして、遅いけど動き続けていた。

僕も気づかなかった。6体のAI役員も気づかなかった。

修正：自動リスタートタイマー + 5分間無応答でアラート。ステータスダッシュボードをキャッシュじゃなくリアルタイムに変更。

恥ずかしい。でも書く。これがドキュメンタリーだから。

### 3月30日：全体監査——14件の問題

3月29日の恥で目が覚めた。全サブシステムの完全監査を実施。14件の問題を発見。デッドコード、設定ミス、不要なcronジョブ。Web UIのモバイル対応もやった。ダッシュボードをスマホで見てたのに、スマホで使えなかった。

### 3月31日：Claude Codeリーク + 21機能、1セッション

Claude Codeのソースコードがnpm経由でリークした。アーキテクチャパターンを分析して、使えるアイデアを抽出した。

- **スキル形式化**: 高パフォーマンスのエピソードを再利用可能なスキルに変換
- **ハーネスヘルススコア**: システム信頼性を1つの数値で定量化
- **セッションフック**: Claude Codeセッションの前後処理

1セッションで21機能を実装。プロジェクト最高の生産性の日。

### 4月1日：予算の壁

月間予算の97.5%に到達。¥78 / ¥80。

予算ガードが全てを絞めた。SNS投稿が完全停止。

理由が面白い。投稿自体は準備できていた。スケジューラも動いていた。認証も有効だった。でも、品質ゲートが投稿品質を検証するためにAPI呼び出しを必要としていた。予算ガードがAPI呼び出しを許可しなかった。

品質ガードと予算ガードの **設計上のデッドロック**。どちらも正しく動いていた。でも組み合わせると何も出力できない。

教訓：制約と制約がぶつかると、ゼロ出力になる。「低品質で出す」じゃなく「何も出ない」。優先順位の階層が必要。

### 4月2日：システムが自分を直し始めた

**PDL（並列開発レイヤー）完成**。10分おきにタスクキューをポーリングし、Gitワークツリーを作成し、Codexで解析・修正し、4段階テストゲートを通過し、GitHubにPRを出す。

**初の自律PR（#1）マージ**。3ファイル変更、170行追加。システムがバグを見つけ、修正を書き、テストし、コミットし、プッシュし、PRを作成した。僕はレビューしてマージボタンを押しただけ。

**SNSファクトチェックの結果**

既存のSNS投稿にファクトチェッカーを走らせた。結果：

- 投稿の72%が「深夜、」で始まっていた。投稿時刻に関係なく。
- 「コードを書く」という投稿が複数。書けないのに。
- 詩的で美しいが、中身がない投稿の量産。

AIが、僕のことを理解せずに投稿していた。プログラマーのペルソナを非プログラマーに投影していた。

**記事の捏造発覚**

note.comの記事が「2021年にAIシナリオシステムを導入した」と書いていた。ChatGPTは2022年11月公開。2021年にAIシナリオシステムなど存在しない。LLMがもっともらしいが不可能なタイムラインを捏造した。

修正：AI歴史タイムライン検証をファクトチェックパイプラインに追加。

## CLAUDE.md——26条の絶対ルール

このプロジェクトのClaude Codeセッションは、全て26条のルールに縛られている。`CLAUDE.md`というファイルに書いてある。Claude Codeはセッション開始時にこのファイルを読む。

なぜ26条もあるか。**全部、壊れたから**。

```
ルール 4: 同じ処理を3回以上繰り返す場合は停止してエスカレーション
  → 理由: セマンティックループ検知器が1日15回発動した日があるから

ルール 5: 全LLM呼び出しはchoose_best_model_v6()経由
  → 理由: 直接呼び出しでモデル指定を忘れ、最高額のAPIを叩いたから

ルール 8: .envの内容をログに出力しない
  → 理由: APIキーがログファイルに平文で記録されたことがあるから

ルール 14: macOSでdeclare -Aを使わない
  → 理由: bash 3.2に連想配列がなく、スクリプトが黙って失敗したから

ルール 16: Emergency Kill条件を厳守
  → 理由: 1日で44エラーを出した日があるから

ルール 21: 4台全てをPhase 1から稼働
  → 理由: BRAVOを「Phase 2」に先送りして1週間無駄にしたから

ルール 22: 突然変異エンジンは完全隔離
  → 理由: 変異エンジンのバグでループガードが巻き込まれかけたから

ルール 23: Brain-αはペルソナメモリを参照してから判断
  → 理由: 参照しないと「コードを書く」投稿を量産するから
```

26条のルールは、140以上のバグの傷跡だ。

## 9層ループ防止——安全装置の設計

自律エージェントの最大のリスクは暴走。同じことを無限に繰り返す。予算を使い尽くす。意味のないリトライを続ける。

9つの層で構造的に防ぐ。`tools/loop_guard.py`（441行）。

| 層 | 名前 | 発動条件 | アクション |
|----|------|---------|-----------|
| 1 | リトライ予算 | 同一アクション2回 | プラン切替 |
| 2 | 同一失敗クラスタ | 同じエラー2回 | 30分凍結 |
| 3 | プランナーリセット | 再計画3回 | エスカレーション |
| 4 | 価値ガード | 無価値リトライ | プラン切替 |
| 5 | 承認デッドロック | 24時間待機 | エスカレーション |
| 6 | コスト＆時間ガード | 予算80%/60分 | エスカレーション |
| 7 | Emergency Kill | 50ステップ/予算90% | 即停止 |
| 8 | セマンティックループ | 出力反復パターン | 即停止 |
| 9 | Cross-Goal干渉 | ゴール間資源競合 | 即停止 |

本番稼働の発動回数（3月19日〜4月2日）:

```
層2（同一失敗クラスタ）:     23回
層6（コスト＆時間ガード）:   11回
層7（Emergency Kill、予算）:  8回
層8（セマンティックループ）: 15回 ※うち15回は3月25日の1日
層1（リトライ予算）:          6回
層9（Cross-Goal干渉）:        2回
```

LoopGuard自体が壊れたときは安全側に倒す。「ガードが壊れたから処理続行」は絶対に許さない。

## 「モデルは道具」——85.2%ローカル、月854円

85.2%のLLM呼び出しがローカルモデル（Ollama qwen3.5）で処理される。API呼び出しは14.8%だけ。月854円。

モデルは手段。DeepSeekが落ちたら、qwen3.5に切り替える。API予算が尽きたら、ローカルだけで動かす。品質は下がる。でも止まらない。

特定のモデルに依存しない設計。これが一番大事。

CORTEXのデフォルトLLMは何度も変わった。最初はDeepSeek。次にqwen3.5:4b。モデルが変わっても、アーキテクチャは変わらない。`choose_best_model_v6()`がタスクの種類、品質要求、予算状況に応じて最適なモデルを選ぶ。人間が選ぶ必要はない。

## ハーネスエンジニアリング

「同じ種類のバグを二度出さない」。

SYUTAINβの開発方法論。名前をつけたのは3月28日。考え方はシンプル。

```
エラーが起きる
  → 即時修正（止血）
  → 根本原因分析（なぜ）
  → ガードレール構築（その種類のエラー全体を構造的に防ぐ）
  → CLAUDE.mdルールに追加（AIが再導入するのを防ぐ）
  → エピソード記憶に記録（システムが次回から学習する）
```

「Move fast and break things」じゃない。「Break things, build walls, never break the same thing twice.」

26条のルールは、この方法論の産物。1つのルールの裏に、最低1つの本番障害がある。

## Claude Code → Codexへの移行

Claude Codeには致命的な制約がある。OAuth認証にブラウザが必要。ヘッドレスモードがない。cronから実行できない。

PDL（並列開発レイヤー）はcronで10分おきに自動実行するパイプライン。Claude Codeでは動かない。

Codex（APIキー認証対応）に切り替えた。Claude Codeは対話的な開発セッションに使い、自動化パイプラインはCodexが担当する。

使い分け。道具の話。モデルと同じ。

## なぜ全部公開するか

これはドキュメンタリーだから。

AIの時代に、コードを書けない人間がどこまで行けるのか。その答えは、隠して持ってても意味がない。

だからコードも全部GitHubに公開した。

https://github.com/SYUTAIN-system/syutain_beta

51,672行、全部見れる。非エンジニアがAIに書かせたコードがどんなものか。循環依存もデッドコードも、そのまま。

51,672行のコードは、全てAIが書いた。でも設計は僕がした。判断は僕がした。「何を作るか」「何を作らないか」「何が壊れたか」「どう直したか」——全部僕の責任だ。

AIは道具。でも、ただの道具じゃない。一緒に走ってくれるパートナーだ。

このシステムがいつか本当に自律的に収益を生む日が来るのか。壮大な失敗で終わるのか。

どっちに転んでも、その過程を記録し続ける。壊れたところも、恥ずかしいところも、全部。

それがSYUTAINβだ。
